## okwud.build_request emits XSD-invalid Dokument (namespace + element-shape bugs)

- **Area**: `okwud`
- **Severity**: `blocker` — the build path is the *only* part of OK-WUD that should work offline; it does not produce a schema-valid envelope, so the `okwud.build_request` → sign → submit flow described in the README cannot pass `okwud_2020_12_29.xsd` even before SOAP/auth concerns.
- **Status of evidence**: confirmed (one validator-confirmed error + six code-review-confirmed structural mismatches).
- **Summary**: `okwud.build_request` returns `{xml, podmioty_count}` cleanly for a court (`kod=S`) request, but the produced XML fails XSD validation against the bundled `src/mcp_zus/okwud/schemas/okwud_2020_12_29.xsd`. Validator stops at a namespace mismatch on `OpisDokumentu`; code review of `src/mcp_zus/okwud/builder.py` plus `models.py` reveals five further structural bugs that would surface once the namespace bug is fixed, including an entirely fictional `ZakresZadanychDanychTyp` shape and silent data loss when more than one podmiot is supplied.
- **Reproduction**:
  ```bash
  python scripts/mcp_call.py call okwud.build_request - < /tmp/okwud_court_input.json
  # then validate with lxml against src/mcp_zus/okwud/schemas/okwud_2020_12_29.xsd
  # full payload + script: evidence/tools/okwud.build_request.json
  #                        evidence/tools/okwud.build_request.xsd_validation.json
  ```
- **Observed**:
  - **Validator output (first error)**:
    ```
    Element '{http://crd.gov.pl/xml/schematy/struktura/2009/11/16/}OpisDokumentu':
      This element is not expected.
      Expected is ( {http://crd.gov.pl/wzor/2020/12/29/10229/}OpisDokumentu ).
    ```
  - **Code-review-confirmed mismatches** (against XSD `okwud_2020_12_29.xsd`):
    1. **Namespace bug**: `builder.py` uses `_qn("OpisDokumentu", "str")` and `_qn("DaneDokumentu", "str")` (struktura NS) but XSD declares them as locally-scoped elements in target NS `http://crd.gov.pl/wzor/2020/12/29/10229/`; only their *types* live in struktura.
    2. **`DaneIdentyfikacyjneWnioskodawcy` element order is wrong**: builder writes `NazwaSkrocona, NIP, REGON`; XSD `DaneIdentyfikacyjneWnioskodawcyTyp` requires sequence `NIP, REGON, NazwaSkrocona, NazwaPelna`.
    3. **`DaneIdentyfikacyjneKontaUdostepnioneDane` element order is wrong**: `_build_podmiot()` writes `ImiePierwsze, Nazwisko, PESEL, RodzajDokumentuTozsamosci, SeriaINumerDokumentu, DataUrodzenia, NazwaSkrocona, NazwaPelna, NIP, REGON`; XSD requires `NIP, REGON, PESEL, RodzajDokumentuTozsamosci, SeriaINumerDokumentu, NazwaSkrocona, NazwaPelna, Nazwisko, ImiePierwsze, DataUrodzenia`.
    4. **`ZakresZadanychDanych` is fundamentally wrong**: pydantic model `ZakresZadanychDanych` and the builder both emit `<DaneIdentyfikacyjne>true</…>`, `<OkresyUbezpieczenia>true</…>`, `<PodstawyWymiaru>true</…>`, `<Swiadczenia>true</…>`, `<Skladki>true</…>`, `<Inne>…</Inne>`. **None** of those element names exist in XSD `ZakresZadanychDanychTyp`, which requires four sub-blocks: `DaneAdrOrazInfoNaTematZgloszDoUb` (boolean leaves: `Adresy, TelefonyKontaktowe, Rachunki, RodzajeTytulyOkresyUbezpDaneEwidAdrIdent, DaneRaportowRozliczeniowych, InformacjaOZgonie`), `InfoUzupOPlatnikachSkladekOsoby` (`DaneEwidAdrWszystkichAktualnychPlatnikow`), `SwiadczIZasilPobieranePrzezOsobe` (boolean leaves: `InformacjeOSwiadczeniach, RachunekOstatniaWyplataSwiadczenia, AdresPocztowyOstatniPrzekazSwiadczenia`), and `InneDane` (string ≤1000 chars).
    5. **Multi-podmioty data-loss / cardinality bug**: builder loops `for podmiot in request.podmioty` emitting one `DaneIdentyfikacyjneKontaUdostepnioneDane` + optional `DaneAdresoweKontaUdostepnioneDane` per subject directly inside `TrescDokumentu`, but the XSD declares both with default `maxOccurs="1"` — a request with N podmioty produces N copies of elements the schema allows only once.
    6. **`sygnatura_sprawy` is silently dropped**: `PodmiotUdostepnioneDane.sygnatura_sprawy` is accepted in the input model but never appears in the output XML; the natural target is the XSD's `OdwolanieDoNumeruSprawy` element which the builder never emits.
- **Expected**: `evidence/readme_claims.md` row A2 (and PROMPT.md exit criterion §3) say the OK-WUD build path should produce a schema-valid envelope (the offline portion is supposed to be ✅ even when live SOAP is ❌). Currently it produces an envelope that the bundled `okwud_2020_12_29.xsd` rejects, with structural bugs that go beyond the validator's first error.
- **Suggested fix / next step**:
  - Fix the namespace of `OpisDokumentu`/`DaneDokumentu` (target NS, not `str:`).
  - Reorder element emission to match XSD sequences in `DaneIdentyfikacyjneWnioskodawcyTyp` and `DaneIdentyfikacyjneDlaUdostepnianychDanychTyp`.
  - Replace `ZakresZadanychDanych` (model + builder) with the four sub-blocks the XSD actually defines; expose the boolean leaves directly so an LLM has a real surface.
  - Decide on a wire layout for multiple podmioty: either enforce `len(podmioty) == 1` in the model (matching XSD reality) or extend the schema to allow `maxOccurs="unbounded"` (out of scope here — would need ZUS spec confirmation).
  - Map `sygnatura_sprawy` to `OdwolanieDoNumeruSprawy`, or document it as deliberately ignored.
  - Embed a `validation: ValidationReport` field in the response (parity with KEDU builders) and/or expose a sibling `okwud.validate` MCP tool.

## okwud.build_request input lacks NIP/REGON/PESEL checksum validators (parity gap with KEDU)

- **Area**: `okwud`
- **Severity**: `minor`
- **Status of evidence**: confirmed (code review of `src/mcp_zus/okwud/models.py`).
- **Summary**: `kedu.build_dra` rejects bad NIP and `kedu.build_zua` rejects bad PESEL via pydantic `field_validator` checksum guards. `okwud.build_request`'s nested models (`DaneIdentyfikacyjneWnioskodawcy`, `PodmiotUdostepnioneDane`) constrain only `max_length` — no checksum validators on `nip`, `regon`, or `pesel`. The OK-WUD XSD itself only enforces digit-count patterns, so a downstream ZUS server would catch a bad value, but in offline use the tool happily accepts payloads with arithmetically-invalid identifiers.
- **Reproduction**: feed `regon:"000000000"` or `pesel:"44051401358"` (bad checksum) to `okwud.build_request` — both pass; compare with `kedu.build_dra` which rejects equivalent bad NIP at parse time.
- **Observed**: no validation error; XML is built with the bad identifier embedded.
- **Expected**: parity with KEDU validators (or an explicit decision to defer checksum validation to ZUS).
- **Suggested fix / next step**: lift the existing `_check_nip` / `_check_pesel` validators from `kedu/models.py` into a shared helper module and reuse on `okwud/models.py`.
